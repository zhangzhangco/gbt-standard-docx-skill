# GB/T 1.2 采标规则起步说明

这个文档不是 `GB/T 1.2` 全量条文摘录，而是面向 skill 实现的最小规则骨架。

## 当前已经明确的判断口径

1. `IDT / MOD / NEQ` 不能只看主观意图，要看正文、结构和附录处理后的实际结果。
2. 删除国际文件中的正文条款，通常不再属于纯编辑性处理，需要进入技术差异说明。
3. 前言必须说明采标方式，并在需要时说明结构调整、技术差异及其原因。
4. 采标场景下，不应把国际文件中的正文技术内容随意删薄；除明确列出的技术差异外，应尽量保留技术实质。
5. 引言、前言、附录体系和规范性引用文件清单允许按我国标准体例调整，但要和采标方式保持一致。

## 适合落成自动检查的项目

1. 前言是否写明 `IDT / MOD / NEQ`
2. `MOD` 情况下是否说明结构调整和技术差异
3. 是否存在“前言写 `IDT`，正文实际有删改”的矛盾
4. 是否删除了正文技术内容但未在技术差异中说明
5. 引言是否保留了不符合我国体例的等同性表述

## 当前建议的实现顺序

1. 先做输入级一致性检查
2. 再做前言和附录完整性检查
3. 最后做基于正文比对的更深层技术差异审查

## 当前已落地到脚本的最小输入约定

1. 顶层可选 `adoption` 对象
2. `adoption.mode` 取值为 `IDT / MOD / NEQ`
3. `adoption.source_standard` 记录原始国际文件
4. `MOD` 场景下，`adoption.structure_adjustments` 和 `adoption.technical_differences` 应为非空
5. 附录可选 `role`
6. 当前约定的附录角色包括：
   - `structure_adjustment`
   - `technical_differences`
   - `correspondence`
   - `other`
