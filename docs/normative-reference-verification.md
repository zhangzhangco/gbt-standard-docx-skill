# 规范性引用文件版本核验

这个能力用于回答一个更具体的问题：规范性引用文件不仅要“写得像”，还要判断是否需要核验其当前状态。

## 规则边界

按 `GB/T 1.1`：

1. `注日期引用` 只适用该日期对应版本。
2. `不注日期引用` 适用最新版本，包括所有修改单。
3. 对于不注日期引用，如果最新版已经不包含被引内容，则包含该内容的最后版本适用。

因此，版本核验的目标不是“默认替换成最新版”，而是回答以下问题：

1. 该引用当前是注日期还是不注日期。
2. 该写法是否符合 `GB/T 1.1`。
3. 该文件当前是否已有新版本、被废止、被历史化，或属于动态注册表。
4. 如果从旧版改到新版，会不会构成技术差异。

## skill 应明确告诉代理做什么

当文稿包含规范性引用文件时，代理不应只检查格式，还应执行一次“版本状态核验”。最低要求包括：

1. 区分 `dated` 与 `undated` 引用。
2. 对 `undated` 引用核验其当前有效版本或当前注册表状态。
3. 对 `dated` 引用核验其是否已被后续版本取代、废止或历史化。
4. 输出结论时区分三类建议：
   - `可保持原样`
   - `建议改为不注日期引用`
   - `如升级版本将构成技术差异，需人工确认`

## 哪些条目必须联网核验

以下情况不应仅靠本地记忆判断：

1. 国际标准、行业标准、团体标准版本号是否已有更新。
2. RFC 是否已被 `Obsoleted`、`Updated`、转为 `Historic`。
3. IANA、W3C、注册表、在线分配表等动态资源的当前状态。
4. 采标项目中的外部标准是否仍处于现行状态。

## 当前仓库中的实现策略

1. `validate_gbt.py`
   负责格式和输入一致性，不负责联网查新。
2. `scripts/check_normative_refs.py`
   负责抽取规范性引用文件并标出哪些条目需要在线版本核验。
3. 联网核验结果应由代理或后续脚本结合权威网站完成，不应在没有核验的情况下自动追新。
